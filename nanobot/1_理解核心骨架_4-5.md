## 4. Tool 基类 (`agent/tools/base.py`)
### 4.1 整体定位
`Tool` 是所有工具的**抽象基类**。nanobot 的每一个能力（读文件、执行命令、搜索网页……）都是一个 Tool 子类。它定义了四个必须实现的接口 + 三个内置的通用能力：
```
必须实现（抽象方法）：
  ├── name        → 工具名（如 "read_file"）
  ├── description → 工具描述（LLM 看到的说明文字）
  ├── parameters  → JSON Schema 格式的参数定义
  └── execute()   → 实际执行逻辑
内置通用能力：
  ├── cast_params()    → 参数类型自动转换
  ├── validate_params() → 参数校验
  └── to_schema()      → 转成 OpenAI function calling 格式
```
### 4.2 四个抽象接口
```python
@property
@abstractmethod
def name(self) -> str: ...        # "read_file"
@property
@abstractmethod
def description(self) -> str: ... # "Read the contents of a file..."
@property
@abstractmethod
def parameters(self) -> dict: ... # JSON Schema 对象
@abstractmethod
async def execute(self, **kwargs) -> str: ...  # 实际执行
```
注意 `name`、`description`、`parameters` 用了 `@property`，意味着子类不用加括号就能访问，像读属性一样。`execute` 是 `async` 的，因为很多工具涉及 IO（读文件、网络请求等）。
### 4.3 `to_schema()` — 转成 OpenAI 格式
```python
def to_schema(self):
    return {
        "type": "function",
        "function": {
            "name": self.name,
            "description": self.description,
            "parameters": self.parameters,
        },
    }
```
这就是 OpenAI function calling 要求的标准格式。`AgentLoop` 每次调 LLM 时，会通过 `ToolRegistry.get_definitions()` 收集所有工具的 schema，一起发给 LLM。
### 4.4 `cast_params()` — 参数类型转换（重点）
**为什么需要这个？** LLM 返回的 tool call 参数类型不一定对。比如 schema 定义 `limit: integer`，但 LLM 可能返回 `"200"`（字符串）而不是 `200`（整数）。`cast_params` 在校验之前先做一轮"宽容转换"。
调用链：
```
cast_params(params)
  └── _cast_object(params, schema)      # 遍历 dict 的每个 key
        └── _cast_value(value, prop_schema)  # 对每个值做类型转换
```
`_cast_value` 的转换规则（按优先级）：

| 目标类型 | 当前类型 | 处理方式 |
|---------|---------|---------|
| `integer` | `int`（非 bool） | 直接返回 |
| `integer` | `str` | 尝试 `int(val)`，失败则原样返回 |
| `number` | `int/float`（非 bool） | 直接返回 |
| `number` | `str` | 尝试 `float(val)`，失败则原样返回 |
| `string` | 任意 | `str(val)`（None 除外） |
| `boolean` | `bool` | 直接返回 |
| `boolean` | `str` | `"true"/"1"/"yes"` → True，`"false"/"0"/"no"` → False |
| `array` | `list` | 递归转换每个元素（如果有 `items` schema） |
| `object` | `dict` | 递归转换每个属性 |
**关键细节：Python 的 `bool` 是 `int` 的子类**
```python
isinstance(True, int)   # True！
isinstance(False, int)  # True！
```
所以代码里到处都在排除 bool：
```python
if target_type == "integer" and isinstance(val, int) and not isinstance(val, bool):
```
如果不排除，`True` 会被当成合法的 integer，这显然不对。
**设计哲学：宽容转换，失败不报错**
注意所有转换失败时都是 `return val`（原样返回），不会抛异常。真正的类型检查交给下一步 `validate_params()`。
### 4.5 `validate_params()` — 参数校验
调用链：
```
validate_params(params)
  └── _validate(params, schema, path="")
        ├── 类型检查（integer/number/string/boolean/array/object）
        ├── enum 检查
        ├── 数值范围（minimum/maximum）
        ├── 字符串长度（minLength/maxLength）
        ├── 必填字段（required）
        └── 递归校验嵌套的 object 和 array
```
返回 `list[str]`（错误列表），空列表表示校验通过。
**`path` 参数的作用**：构建错误路径，方便定位嵌套结构中的问题。
```python
# 顶层参数
_validate({"limit": "abc"}, schema, "")
# → ["parameter.limit should be integer"]
# 嵌套参数
_validate({"config": {"timeout": -1}}, schema, "")
# → ["parameter.config.timeout must be >= 0"]
```
**校验规则一览**：

| 规则 | 适用类型 | 示例 |
|------|---------|------|
| 类型匹配 | 全部 | `integer` 收到 `str` → 报错 |
| `enum` | 全部 | 值不在枚举列表中 → 报错 |
| `minimum` / `maximum` | integer, number | 值超出范围 → 报错 |
| `minLength` / `maxLength` | string | 字符串太长/太短 → 报错 |
| `required` | object | 缺少必填字段 → 报错 |
| 递归 `properties` | object | 对每个属性递归校验 |
| 递归 `items` | array | 对每个元素递归校验 |
### 4.6 一个具体子类的样子
以 `ReadFileTool` 为例（概念示意）：
```python
class ReadFileTool(Tool):
    @property
    def name(self): return "read_file"
    
    @property
    def description(self): return "Read the contents of a file..."
    
    @property
    def parameters(self):
        return {
            "type": "object",
            "properties": {
                "path": {"type": "string", "description": "The file path"},
                "offset": {"type": "integer", "description": "Line number to start"},
                "limit": {"type": "integer", "description": "Max lines to read"},
            },
            "required": ["path"],
        }
    
    async def execute(self, path: str, offset: int = 1, limit: int = 2000) -> str:
        # 实际读文件的逻辑
        ...
```

## 5. ToolRegistry (`agent/tools/registry.py`)
### 5.1 整体定位
`ToolRegistry` 是工具的**注册中心**，负责：
- 存储所有工具实例
- 按名称查找工具
- 收集所有工具的 schema 定义
- 统一执行入口（含类型转换 + 校验 + 错误处理）
数据结构就是一个简单的字典：
```python
self._tools: dict[str, Tool] = {}
```
key 是工具名（如 `"read_file"`），value 是 Tool 实例。
### 5.2 注册与查找
```python
def register(self, tool: Tool):     # 注册
    self._tools[tool.name] = tool
def unregister(self, name: str):    # 注销
    self._tools.pop(name, None)
def get(self, name: str) -> Tool | None:  # 查找
    return self._tools.get(name)
def has(self, name: str) -> bool:   # 是否存在
    return name in self._tools
```
还实现了 `__len__` 和 `__contains__`，支持 `len(registry)` 和 `"read_file" in registry` 的写法。
### 5.3 `get_definitions()` — 收集所有 schema
```python
def get_definitions(self) -> list[dict]:
    return [tool.to_schema() for tool in self._tools.values()]
```
在 `AgentLoop._run_agent_loop()` 中每次调 LLM 前调用：
```python
tool_defs = self.tools.get_definitions()  # 拿到所有工具的 schema
response = await self.provider.chat_with_retry(
    messages=messages,
    tools=tool_defs,    # 传给 LLM
    model=self.model,
)
```
### 5.4 `execute()` — 统一执行入口（重点！）
这是整个工具系统最关键的方法，流程如下：
```
execute(name, params)
  │
  ├─ 1. 查找工具
  │   tool = self._tools.get(name)
  │   找不到 → 返回错误信息 + 可用工具列表
  │
  ├─ 2. 类型转换
  │   params = tool.cast_params(params)
  │   （宽容转换，不报错）
  │
  ├─ 3. 参数校验
  │   errors = tool.validate_params(params)
  │   有错误 → 返回错误信息 + _HINT
  │
  ├─ 4. 执行工具
  │   result = await tool.execute(**params)
  │
  ├─ 5. 检查工具自身返回的错误
  │   result 以 "Error" 开头 → 追加 _HINT
  │
  └─ 6. 异常兜底
      任何未捕获异常 → 返回错误信息 + _HINT
```
### 5.5 `_HINT` 的设计
```python
_HINT = "\n\n[Analyze the error above and try a different approach.]"
```
这个提示语会追加到**所有错误返回**的末尾，然后作为 tool result 发回给 LLM。
为什么这么做？因为 LLM 收到工具报错后，可能会：
- 反复用同样的错误参数重试（死循环）
- 不知道该怎么修正
加了 `_HINT` 后，LLM 会意识到"上一步出错了，我需要换一种方式"，比如：
- `read_file` 路径不对 → 换个路径重试
- `exec` 命令不存在 → 换个命令
- 参数类型不对 → 修正参数
这个设计让 agent 具备了**自我纠错能力**。
### 5.6 错误处理策略总结
| 错误场景 | 处理方式 | 返回给 LLM 的内容 |
|---------|---------|------------------|
| 工具不存在 | 直接返回 | `"Error: Tool 'xxx' not found. Available: ..."` + _HINT |
| 参数校验失败 | 直接返回 | `"Error: Invalid parameters: ..."` + _HINT |
| 工具执行返回 Error | 追加 _HINT | 原始错误信息 + _HINT |
| 工具执行抛异常 | catch 兜底 | `"Error executing xxx: ..."` + _HINT |
**核心原则：永远不抛异常给上层，所有错误都转成字符串返回给 LLM。**
这样 `AgentLoop` 不需要 try-catch 每个工具调用，逻辑更简洁。LLM 拿到错误信息后自行决定下一步。

### 两个类的关系图
```
ToolRegistry
  │
  ├── _tools: {"read_file": ReadFileTool, "exec": ExecTool, ...}
  │
  ├── get_definitions() ──→ [每个 tool.to_schema()]
  │                              │
  │                              └── Tool.to_schema() → OpenAI 格式
  │
  └── execute(name, params)
        │
        ├── Tool.cast_params()   → 宽容类型转换
        ├── Tool.validate_params() → 严格校验
        └── Tool.execute(**params) → 实际执行
```
