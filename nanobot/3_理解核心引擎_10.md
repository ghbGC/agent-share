## 10. AgentLoop (`agent/loop.py`) — nanobot 的大脑
### 10.1 整体定位
AgentLoop 是 nanobot 的**核心引擎**，相当于整个系统的心脏。它：
- 从消息总线接收用户消息
- 组装上下文、调用 LLM、执行工具
- 把响应发回消息总线
所有其他模块（Context、Memory、Skills、Session、Tools）都是为它服务的。

### 10.4 `run()` — 主事件循环
```python
async def run(self):
    self._running = True
    await self._connect_mcp()  # 连接 MCP 服务器（一次性，懒加载）
    while self._running:
        try:
            msg = await asyncio.wait_for(self.bus.consume_inbound(), timeout=1.0)
        except asyncio.TimeoutError:
            continue  # 没消息就继续等
        cmd = msg.content.strip().lower()
        if cmd == "/stop":
            await self._handle_stop(msg)
        elif cmd == "/restart":
            await self._handle_restart(msg)
        else:
            # 普通消息 → 创建异步任务处理
            task = asyncio.create_task(self._dispatch(msg))
            self._active_tasks.setdefault(msg.session_key, []).append(task)
            task.add_done_callback(...)  # 完成后从 _active_tasks 移除
```
**设计要点**：
1. **`timeout=1.0`**：每秒检查一次，不是无限阻塞。这样即使没有消息，也能响应 `/stop` 等命令（虽然 `/stop` 也是通过消息触发的，但 timeout 保证了循环不会卡死）
2. **`asyncio.create_task`**：每条消息创建一个独立的异步任务。这意味着多条消息可以"排队"处理，但不会真正并行（因为有全局锁）
3. **`_active_tasks` 追踪**：按 session_key 分组存储活跃任务，用于 `/stop` 命令取消
4. **`/stop` 和 `/restart` 不经过 `_dispatch`**：它们是同步处理的，不走全局锁，确保即使有任务在跑也能立即响应
### 10.5 `/stop` 和 `/restart` 命令
**`_handle_stop()`**：
```python
async def _handle_stop(self, msg):
    tasks = self._active_tasks.pop(msg.session_key, [])
    cancelled = sum(1 for t in tasks if not t.done() and t.cancel())
    for t in tasks:
        await t  # 等待取消完成
    sub_cancelled = await self.subagents.cancel_by_session(msg.session_key)
    # 返回取消数量
```
取消该 session 的所有活跃任务 + 子 agent。
**`_handle_restart()`**：
```python
async def _handle_restart(self, msg):
    await self.bus.publish_outbound(...)  # 先回复"Restarting..."
    async def _do_restart():
        await asyncio.sleep(1)  # 等回复发出去
        os.execv(sys.executable, ["-m", "nanobot"] + sys.argv[1:])
    asyncio.create_task(_do_restart())
```
用 `os.execv` 替换当前进程，实现原地重启。先等 1 秒确保"Restarting..."消息发出去。
### 10.6 `_dispatch()` — 全局锁 + 异步分发
```python
async def _dispatch(self, msg):
    async with self._processing_lock:  # ← 全局锁！
        try:
            response = await self._process_message(msg)
            if response is not None:
                await self.bus.publish_outbound(response)
            elif msg.channel == "cli":
                # CLI 模式需要空响应来解除阻塞
                await self.bus.publish_outbound(OutboundMessage(
                    channel=msg.channel, chat_id=msg.chat_id,
                    content="", metadata=msg.metadata or {},
                ))
        except asyncio.CancelledError:
            raise  # /stop 取消，不吞异常
        except Exception:
            # 其他异常 → 返回错误消息给用户
            await self.bus.publish_outbound(OutboundMessage(
                channel=msg.channel, chat_id=msg.chat_id,
                content="Sorry, I encountered an error.",
            ))
```
**为什么需要全局锁？**
虽然 `run()` 用 `create_task` 创建了多个任务，但 `_processing_lock` 确保同一时间只有一条消息在被处理。原因：
- LLM 调用是串行的（共享同一个 provider）
- Session 写入需要串行（避免并发写冲突）
- Memory consolidate 需要串行（避免重复归档）
**但 `/stop` 可以打断**：`CancelledError` 被重新抛出，不会被 `except Exception` 吞掉。
**`response is None` 的含义**：当 agent 已经通过 `message` 工具主动发了消息（`mt._sent_in_turn == True`），`_process_message` 返回 None，避免重复发送。
### 10.7 `_process_message()` — 完整消息处理流程（核心中的核心）
这是整个 nanobot 最复杂的方法，处理一条消息的完整生命周期：
```
_process_message(msg)
  │
  ├─ 1. 判断消息来源
  │   ├─ system 消息（子 agent 回报）→ 特殊处理
  │   └─ 普通用户消息 → 正常处理
  │
  ├─ 2. 斜杠命令处理
  │   ├─ /new → 清空 session + 归档旧消息
  │   └─ /help → 返回帮助文本
  │
  ├─ 3. 记忆压缩检查
  │   maybe_consolidate_by_tokens(session)
  │
  ├─ 4. 设置工具上下文
  │   _set_tool_context(channel, chat_id, message_id)
  │   message_tool.start_turn()
  │
  ├─ 5. 构建消息列表
  │   history = session.get_history()
  │   messages = context.build_messages(history, msg.content, media, ...)
  │
  ├─ 6. 运行 agent 循环
  │   final_content, tools_used, all_msgs = _run_agent_loop(messages, on_progress)
  │
  ├─ 7. 保存到 session
  │   _save_turn(session, all_msgs, skip)
  │   sessions.save(session)
  │
  ├─ 8. 后台触发记忆压缩
  │   _schedule_background(maybe_consolidate_by_tokens(session))
  │
  └─ 9. 返回响应
      ├─ 如果 message 工具已发送 → 返回 None
      └─ 否则 → 返回 OutboundMessage(final_content)
```
**system 消息的特殊处理**（第 364-382 行）：
子 agent 完成任务后，通过 bus 注入一条 `channel="system"` 的消息。`_process_message` 解析 `chat_id` 中的 `"channel:chat_id"` 格式，恢复原始的 channel 和 chat_id，然后正常走 agent 循环处理。这样子 agent 的结果会经过主 agent 的自然语言总结后再发给用户。
**`/new` 命令的处理**（第 392-402 行）：
```python
if cmd == "/new":
    snapshot = session.messages[session.last_consolidated:]  # 取未归档的消息
    session.clear()          # 清空 session
    self.sessions.save(session)
    self.sessions.invalidate(session.key)  # 清除内存缓存
    if snapshot:
        self._schedule_background(
            self.memory_consolidator.archive_messages(snapshot)  # 后台归档
        )
    return OutboundMessage(..., content="New session started.")
```
注意：清空前先拍快照，然后**后台归档**，不阻塞用户。
**`on_progress` 回调**（第 429-435 行）：
```python
async def _bus_progress(content: str, *, tool_hint: bool = False):
    meta = {"_progress": True, "_tool_hint": tool_hint}
    await self.bus.publish_outbound(OutboundMessage(
        channel=msg.channel, chat_id=msg.chat_id, content=content, metadata=meta,

```
agent 循环中每次工具调用前，会通过这个回调发送进度消息（LLM 的思考过程 + 工具调用提示）。飞书通道收到 `_progress=True` 的消息后，会以卡片形式展示，让用户看到 agent 在做什么。
### 10.8 `_run_agent_loop()` — LLM 调用 + 工具执行循环
这是 agent 的"思考-行动"循环：
```python
async def _run_agent_loop(self, initial_messages, on_progress=None):
    messages = initial_messages
    iteration = 0
    final_content = None
    tools_used = []
    while iteration < self.max_iterations:  # 最多 40 轮
        iteration += 1
        # 1. 调 LLM
        response = await self.provider.chat_with_retry(
            messages=messages, tools=tool_defs, model=self.model,
        )
        if response.has_tool_calls:
            # 2a. LLM 想调用工具
            # → 发送进度（思考 + 工具提示）
            # → 追加 assistant message（含 tool_calls）
            # → 逐个执行工具 → 追加 tool result
            for tool_call in response.tool_calls:
                result = await self.tools.execute(tool_call.name, tool_call.arguments)
                messages = self.context.add_tool_result(messages, ...)
        else:
            # 2b. LLM 返回最终文本
            if response.finish_reason == "error":
                final_content = clean or "Sorry, I encountered an error..."
                break  # 错误响应不保存到 session
            messages = self.context.add_assistant_message(messages, clean, ...)
            final_content = clean
            break  # 正常结束
    # 3. 超过最大轮次
    if final_content is None and iteration >= self.max_iterations:
        final_content = "I reached the maximum number of tool call iterations..."
    return final_content, tools_used, messages
```
**流程图**：
```
┌──────────────────────────────────────────┐
│           _run_agent_loop                │
│                                          │
│  ┌─────────────────────────────────────┐ │
│  │  while iteration < 40:             │ │
│  │                                     │ │
│  │   调 LLM(messages + tools)         │ │
│  │         │                           │ │
│  │    ┌────┴────┐                      │ │
│  │    │         │                      │ │
│  │  有工具调用  无工具调用              │ │
│  │    │         │                      │ │
│  │    │    返回 final_content          │ │
│  │    │    break                       │ │
│  │    │                                │ │
│  │  执行工具                           │ │
│  │  追加 tool result                   │ │
│  │  继续循环 ──→                       │ │
│  └─────────────────────────────────────┘ │
│                                          │
│  return (final_content, tools_used, msgs)│
└──────────────────────────────────────────┘
```
**关键细节**：
1. **`_strip_think()`**：去掉 LLM 输出中的 `💭...💭` 思考块（DeepSeek、Kimi 等模型会输出这种格式），避免发给用户
2. **`_tool_hint()`**：把工具调用格式化为简洁提示，如 `read_file("path/to/file")`，通过 `on_progress` 发给用户
3. **错误响应不保存**（第 236-239 行）：
   ```python
   if response.finish_reason == "error":
       final_content = clean or "Sorry..."
       break  # 直接 break，不追加到 messages
   ```
   注释说"error responses can poison the context and cause permanent 400 loops"。如果错误响应被保存到 session，下次 LLM 看到错误内容可能继续犯错，形成死循环。
4. **工具是串行执行的**：`for tool_call in response.tool_calls` 逐个执行，不是并行的。这保证了工具之间的依赖关系（比如先读文件再编辑）
5. **`max_iterations=40`**：防止 LLM 无限循环调工具。达到上限后返回提示信息
### 10.9 `_save_turn()` — 消息持久化 + 截断 + 清理
```python
def _save_turn(self, session, messages, skip):
    for m in messages[skip:]:  # skip 之前的消息已经保存过了
        entry = dict(m)
        role, content = entry.get("role"), entry.get("content")
        # 1. 跳过空的 assistant 消息
        if role == "assistant" and not content and not entry.get("tool_calls"):
            continue
        # 2. 截断过长的 tool result
        if role == "tool" and len(content) > 16_000:
            entry["content"] = content[:16_000] + "\n... (truncated)"
        # 3. 清理 user 消息中的 runtime context
        elif role == "user":
            if isinstance(content, str) and content.startswith("[Runtime Context"):
                # 去掉 runtime context 前缀，只保留用户文本
                parts = content.split("\n\n", 1)
                entry["content"] = parts[1] if len(parts) > 1 else ...
            if isinstance(content, list):
                # 多模态消息：去掉 runtime context 文本块
                # base64 图片 → 替换为 "[image]" 占位符
                ...
        entry.setdefault("timestamp", datetime.now().isoformat())
        session.messages.append(entry)
```
**`skip` 参数的含义**：
`messages` 列表包含 `[system, ...history, user, assistant, tool, assistant, ...]`。`skip = 1 + len(history)` 表示跳过 system prompt 和历史消息（它们已经在之前的 turn 中保存过了），只保存本轮新增的消息。
**三种清理操作**：

| 操作 | 原因 |
|------|------|
| 跳过空 assistant | 空 content + 无 tool_calls 的消息会污染 session 上下文 |
| 截断 tool result（16KB） | 防止单个工具输出（如读大文件）占用过多 token |
| 去掉 runtime context | 这是临时元数据，不需要持久化 |
| base64 图片 → `[image]` | 图片的 base64 数据很大，持久化毫无意义 |
### 10.10 `_set_tool_context()` — 工具上下文注入
```python
def _set_tool_context(self, channel, chat_id, message_id=None):
    for name in ("message", "spawn", "cron"):
        if tool := self.tools.get(name):
            if hasattr(tool, "set_context"):
                tool.set_context(channel, chat_id, ...)
```
每次处理消息前，把当前的 channel 和 chat_id 告诉 `message`、`spawn`、`cron` 工具。这样：
- `MessageTool` 知道往哪发消息
- `SpawnTool` 知道子 agent 完成后通知谁
- `CronTool` 知道定时任务触发后发给谁
### 10.11 `_schedule_background()` — 后台任务调度
```python
def _schedule_background(self, coro):
    task = asyncio.create_task(coro)
    self._background_tasks.append(task)
    task.add_done_callback(self._background_tasks.remove)
```
记忆压缩等耗时操作被调度为后台任务，不阻塞用户响应。`_background_tasks` 列表追踪这些任务，在 `close_mcp()` 时会等待它们全部完成。
### 10.12 `process_direct()` — 直接处理入口
```python
async def process_direct(self, content, session_key="cli:direct", ...):
    await self._connect_mcp()
    msg = InboundMessage(channel=channel, sender_id="user", chat_id=chat_id, content=content)

```
这是给 CLI 模式和 Cron 使用的入口。构造一个虚拟的 `InboundMessage`，然后走正常的 `_process_message` 流程。
