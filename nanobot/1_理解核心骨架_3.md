> **3. Session** (`session/manager.py`) — Session 数据结构 + JSONL 持久化 + `get_history()` 的合法边界对齐逻辑
## Session 模块
两个类：**Session**（数据）和 **SessionManager**（管理）
### 一、Session（数据类）
#### 1.1 数据结构
```python
@dataclass
class Session:
    key: str                    # 会话唯一标识，格式 "channel:chat_id"
    messages: list[dict]        # 所有消息的列表（append-only）
    created_at: datetime        # 创建时间
    updated_at: datetime        # 最后更新时间
    metadata: dict              # 额外元数据（目前没看到具体用途）
    last_consolidated: int = 0  # 已归档的消息数量（偏移量）
```
**关键设计思想：messages 是 append-only（只追加不修改）**
为什么？因为 LLM 有 prompt cache 机制。如果消息列表不变，前缀部分可以命中缓存，节省 token 和延迟。归档操作不删除消息，只是移动 `last_consolidated` 指针。
#### 1.2 `add_message()` — 添加消息
```python
def add_message(self, role, content, **kwargs):
    msg = {"role": role, "content": content, "timestamp": ..., **kwargs}
    self.messages.append(msg)
    self.updated_at = datetime.now()
```
很简单，就是往列表末尾追加。`**kwargs` 允许传入额外字段，比如 `tool_calls`、`tools_used` 等。
#### 1.3 `get_history()` — 获取 LLM 需要的历史
这是 Session 最复杂的方法，做了**三步裁剪**：
```
全部消息
  │
  ├─ 第一步：去掉已归档的部分
  │   unconsolidated = messages[last_consolidated:]
  │
  ├─ 第二步：限制最大条数
  │   sliced = unconsolidated[-max_messages:]
  │
  ├─ 第三步：去掉开头的非 user 消息
  │   （避免以 assistant/tool 开头，LLM 会困惑）
  │
  ├─ 第四步：对齐 tool-call 边界（_find_legal_start）
  │   （避免孤立的 tool result 没有对应的 assistant tool_calls）
  │
  └─ 第五步：只保留 LLM 需要的字段
      （role, content, tool_calls, tool_call_id, name）
```
**为什么需要第四步？** 举个例子：
```
messages[0]: assistant (tool_calls: [id=A, id=B])  ← 被截掉了
messages[1]: tool (tool_call_id=A)                  ← 孤儿！
messages[2]: tool (tool_call_id=B)                  ← 孤儿！
messages[3]: user "继续"
```
如果 `max_messages=3` 截掉了第 0 条，LLM 会收到两个没有上下文的 tool result，大多数 provider 会报错。`_find_legal_start()` 就是解决这个问题。
#### 1.4 `_find_legal_start()` — 找到合法的起始位置
```python
@staticmethod
def _find_legal_start(messages):
    declared: set[str] = set()  # 已声明的 tool_call id
    start = 0
    for i, msg in enumerate(messages):
        if role == "assistant":
            # 记录这个 assistant 声明了哪些 tool_call
            for tc in msg.get("tool_calls") or []:
                declared.add(tc["id"])
        elif role == "tool":
            tid = msg.get("tool_call_id")
            if tid not in declared:
                # 发现孤儿 tool result！
                # 把 start 移到这个 orphan 之后，重新开始计数
                start = i + 1
                declared.clear()
                # 重新扫描 start 之后的 assistant，重建 declared 集合
                for prev in messages[start:i+1]:
                    if prev["role"] == "assistant":
                        for tc in prev.get("tool_calls") or []:
                            declared.add(tc["id"])
    return start
```
**用例子理解：**
```
index 0: assistant (tool_calls: [A, B])
index 1: tool (call_id=A)     ✓ A 在 declared 中
index 2: tool (call_id=B)     ✓ B 在 declared 中
index 3: user "你好"
index 4: assistant (tool_calls: [C])
index 5: tool (call_id=C)     ✓ C 在 declared 中
```
这种情况 `start=0`，没有孤儿，全部合法。
```
index 0: tool (call_id=X)     ✗ X 不在 declared 中！→ start=1, declared 清空
index 1: assistant (tool_calls: [Y])
index 2: tool (call_id=Y)     ✓
```
这种情况 `start=1`，跳过了开头的孤儿。
```
index 0: assistant (tool_calls: [A])
index 1: tool (call_id=A)     ✓
index 2: tool (call_id=B)     ✗ B 不在 declared 中！→ start=3, declared 清空
index 3: user "继续"
```
这种情况 `start=3`，跳过了 index 0-2（因为 B 是孤儿，连带它前面的 assistant 和 A 也一起丢弃，保证不出现半截对话）。
#### 1.5 `clear()` — 清空会话
```python
def clear(self):
    self.messages = []
    self.last_consolidated = 0
```
对应 `/new` 命令，开始全新对话。

### 二、SessionManager（管理类）
#### 2.1 存储路径设计
```
workspace/sessions/
  feishu_ou_xxx.jsonl   ← 当前版本
  telegram_xxx.jsonl
~/.nanobot/sessions/                                    ← 旧版（legacy）
  feishu_xxx.jsonl                                      ← 会自动迁移
```
key 格式是 `channel:chat_id`，存文件时把 `:` 替换成 `_`，再用 `safe_filename()` 去掉不安全字符。
#### 2.2 JSONL 文件格式
```jsonl
{"_type":"metadata","key":"feishu:ou_xxx","created_at":"...","updated_at":"...","metadata":{},"last_consolidated":10}
{"role":"user","content":"你好","timestamp":"2026-03-21T16:00:00"}
{"role":"assistant","content":"你好！","timestamp":"2026-03-21T16:00:01","tools_used":["read_file"]}
{"role":"user","content":"帮我读个文件","timestamp":"2026-03-21T16:00:05"}
...
```
第一行是元数据（`_type: metadata`），后面每行一条消息。这种格式的好处：
- **追加写入高效**（不用每次重写整个文件）
- **人类可读**（直接 cat 就能看）
- **容错性好**（某行坏了不影响其他行）
但实际上 `save()` 是全量重写的（`open(path, "w")`），不是追加。这是因为 `last_consolidated` 等元数据可能变化，需要更新第一行。
#### 2.3 `get_or_create()` — 获取或创建会话
```python
def get_or_create(self, key):
    if key in self._cache:      # 先查内存缓存
        return self._cache[key]
    session = self._load(key)   # 再从磁盘加载
    if session is None:
        session = Session(key=key)  # 都没有就新建
    self._cache[key] = session
    return session
```
**两级缓存**：内存 `_cache` → 磁盘 JSONL 文件。


### 三、Session 在整体流程中的位置
```
用户发消息
    │
    ▼
AgentLoop._process_message(msg)
    │
    ├─ session = sessions.get_or_create(msg.session_key)  ← 获取会话
    │
    ├─ history = session.get_history()                     ← 获取历史
    │
    ├─ messages = context.build_messages(history, ...)     ← 构建完整 prompt
    │
    ├─ ... LLM 调用 + 工具执行 ...
    │
    ├─ session.add_message("user", ...)                   ← 保存用户消息
    ├─ session.add_message("assistant", ..., tools_used=...)  ← 保存助手回复
    │
    ├─ sessions.save(session)                             ← 持久化到磁盘
    │
    └─ memory_consolidator.maybe_consolidate_by_tokens(session)  ← 后台检查是否需要压缩
```

### 四、核心设计总结
| 设计点 | 说明 |
|--------|------|
| **append-only** | 消息只追加不删除，利于 LLM prompt cache |
| **last_consolidated 偏移量** | 归档不删消息，只移动指针，`get_history()` 自动跳过已归档部分 |
| **tool-call 边界对齐** | `_find_legal_start()` 确保不会出现孤立的 tool result |
| **JSONL 格式** | 人类可读、逐行解析容错好 |
| **内存+磁盘两级缓存** | `get_or_create()` 先查内存再查磁盘 |
| **legacy 迁移** | 自动从旧路径迁移会话文件 |
| **字段过滤** | `get_history()` 只返回 LLM 需要的字段，不暴露 timestamp 等内部数据 |
