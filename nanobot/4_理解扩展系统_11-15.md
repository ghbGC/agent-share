> **11. SubagentManager** (`agent/subagent.py`) — 独立工具集 + 独立循环 + bus 回报机制
> **12. LLMProvider** (`providers/base.py`) — 抽象接口 + `chat_with_retry` 重试策略 + 图片降级
> **13. CronService** (`cron/service.py`) — 三种调度模式 + 定时器 + 磁盘持久化

## 11. SubagentManager (`agent/subagent.py`)
### 11.1 设计动机
主 AgentLoop 有全局锁，同一时间只能处理一条消息。如果某个任务很耗时（比如批量处理文件），会阻塞其他消息。子 agent 解决这个问题——它是一个**独立的 asyncio.Task**，和主 agent 并行运行，完成后通过 bus 汇报结果。
### 11.2 子 agent vs 主 agent 的区别
| | 主 AgentLoop | 子 agent |
|---|---|---|
| 工具集 | 全部 10 个 | 7 个（去掉 message、spawn、cron） |
| 最大轮次 | 40 | 15 |
| system prompt | 完整（identity + bootstrap + memory + skills） | 精简（只有身份 + workspace + skills 摘要） |
| 会话管理 | 有 Session 持久化 | 无，一次性 |
| 进度通知 | 有 on_progress 回调 | 无 |
| 结果去向 | 直接返回给用户 | 通过 bus 注入，主 agent 总结后转发 |
**为什么去掉 message、spawn、cron？**
- **message**：子 agent 不应该直接给用户发消息，结果需要经过主 agent 过滤
- **spawn**：防止无限递归（子 agent 再生子 agent）
- **cron**：定时任务管理应该由主 agent 统一控制
### 11.3 `spawn()` — 创建子 agent
```python
async def spawn(self, task, label, origin_channel, origin_chat_id, session_key):
    task_id = str(uuid.uuid4())[:8]  # 8 位短 ID
    bg_task = asyncio.create_task(self._run_subagent(...))
    self._running_tasks[task_id] = bg_task
    self._session_tasks[session_key].add(task_id)
    bg_task.add_done_callback(_cleanup)  # 完成后自动清理
    return f"Subagent [{label}] started (id: {task_id}). I'll notify you when it completes."
```
**两个索引结构**：
```python
_running_tasks: dict[str, asyncio.Task]       # task_id → Task（快速取消）
_session_tasks: dict[str, set[str]]           # session_key → {task_id, ...}（按会话批量取消）
```
`_cleanup` 回调在 Task 完成后自动从两个字典中移除，不会内存泄漏。
### 11.4 `_run_subagent()` — 子 agent 的独立循环
```python
async def _run_subagent(self, task_id, task, label, origin):
    try:
        # 1. 构建独立工具集
        tools = ToolRegistry()
        tools.register(ReadFileTool(...))
        tools.register(WriteFileTool(...))
        # ... 没有 MessageTool、SpawnTool、CronTool
        # 2. 构建精简 system prompt
        system_prompt = self._build_subagent_prompt()
        # 3. 初始消息
        messages = [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": task},
        ]
        # 4. 独立的 agent 循环（最多 15 轮）
        while iteration < 15:
            response = await self.provider.chat_with_retry(...)
            if response.has_tool_calls:
                # 执行工具，追加结果
            else:
                final_result = response.content
                break
        # 5. 汇报结果
        await self._announce_result(task_id, label, task, final_result, origin, "ok")
    except Exception as e:
        await self._announce_result(..., "error")
```
注意子 agent 的循环是**手写的简化版**，没有复用 AgentLoop._run_agent_loop()。这意味着子 agent 没有：
- 进度通知
- 思考块清理（_strip_think）
- 错误响应过滤
### 11.5 `_announce_result()` — bus 回报机制（核心设计）
```python
async def _announce_result(self, task_id, label, task, result, origin, status):
    announce_content = f"""[Subagent '{label}' {status_text}]
Task: {task}
Result:
{result}
Summarize this naturally for the user. Keep it brief (1-2 sentences).
Do not mention technical details like "subagent" or task IDs."""
    msg = InboundMessage(
        channel="system",           # ← 特殊标记
        sender_id="subagent",
        chat_id=f"{origin['channel']}:{origin['chat_id']}",  # ← 编码原始路由信息
        content=announce_content,
    )
    await self.bus.publish_inbound(msg)
```
**巧妙之处**：
1. 子 agent 不直接发消息给用户，而是往 inbound 队列注入一条 `channel="system"` 的消息
2. `chat_id` 编码了原始的 `channel:chat_id`（如 `"feishu:ou_xxx"`）
3. AgentLoop._process_message() 识别到 system 消息后，解析 chat_id 恢复原始路由
4. 主 agent 看到子 agent 的结果，自然语言总结后发给用户
**为什么这样设计？**
- 子 agent 的结果经过主 agent 的"翻译"，更自然
- 主 agent 可以决定是否需要通知用户（结合 Evaluator）
- 统一走消息总线，不需要额外的通信通道
### 11.6 `_build_subagent_prompt()` — 精简的 system prompt
```python
def _build_subagent_prompt(self):
    parts = [f"""# Subagent
{time_ctx}
You are a subagent spawned by the main agent to complete a specific task.
Stay focused on the assigned task.
## Workspace
{self.workspace}"""]
    skills_summary = SkillsLoader(self.workspace).build_skills_summary()
    if skills_summary:
        parts.append(f"## Skills\n\n{skills_summary}")
    return "\n\n".join(parts)
```
对比主 agent 的 system prompt，子 agent **没有**：
- Bootstrap Files（AGENTS.md、SOUL.md 等）
- Memory（MEMORY.md）
- always 技能的完整内容
子 agent 是"无状态的"，不继承主 agent 的记忆和个性。它只需要知道 workspace 路径和可用技能。

## 12. LLMProvider (`providers/base.py`)
### 12.1 数据结构
```python
@dataclass
class ToolCallRequest:
    id: str
    name: str
    arguments: dict[str, Any]
    # provider_specific_fields: 用于传递 provider 特有的字段
@dataclass
class LLMResponse:
    content: str | None
    tool_calls: list[ToolCallRequest]
    finish_reason: str  # "stop" | "tool_calls" | "error" | "length"
    usage: dict[str, int]
    reasoning_content: str | None   # DeepSeek-R1、Kimi 的思考链
    thinking_blocks: list[dict]     # Anthropic 的扩展思考
@dataclass(frozen=True)
class GenerationSettings:
    temperature: float = 0.7
    max_tokens: int = 4096
    reasoning_effort: str | None = None
```
**`finish_reason` 的含义**：

| 值 | 含义 |
|---|---|
| `"stop"` | 正常结束 |
| `"tool_calls"` | LLM 想调用工具 |
| `"length"` | 达到 max_tokens 上限 |
| `"error"` | 调用出错（网络、API 限制等） |
### 12.2 抽象接口
```python
class LLMProvider(ABC):
    @abstractmethod
    async def chat(self, messages, tools, model, max_tokens, temperature, 
                   reasoning_effort, tool_choice) -> LLMResponse: ...
    @abstractmethod
    def get_default_model(self) -> str: ...
```
子类需要实现这两个方法。目前有：
- `LiteLLMProvider` — 通过 litellm 库支持 100+ 模型
- `AzureOpenAIProvider` — Azure OpenAI 专用
- `CustomProvider` — 自定义 provider
### 12.3 `chat_with_retry()` — 重试策略（核心方法）
```python
_CHAT_RETRY_DELAYS = (1, 2, 4)  # 指数退避：1s, 2s, 4s
```
完整流程：
```
chat_with_retry(messages, tools, model, ...)
  │
  ├─ 1. 参数默认值填充
  │   max_tokens 未传 → 用 self.generation.max_tokens
  │   temperature 未传 → 用 self.generation.temperature
  │
  ├─ 2. 重试循环（最多 3 次）
  │   │
  │   ├─ attempt 1: _safe_chat(**kw)
  │   │   │
  │   │   ├─ finish_reason != "error" → 直接返回 ✅
  │   │   │
  │   │   ├─ finish_reason == "error" + 非瞬态错误
  │   │   │   ├─ 是图片不支持错误？
  │   │   │   │   └─ 去掉图片，重试一次 → 返回结果
  │   │   │   └─ 不是 → 直接返回错误（不重试）
  │   │   │
  │   │   └─ finish_reason == "error" + 瞬态错误（429/500/502/503/504）
  │   │       └─ sleep(1s) → 继续循环
  │   │
  │   ├─ attempt 2: _safe_chat(**kw)
  │   │   └─ 同上逻辑，sleep(2s)
  │   │
  │   └─ attempt 3: _safe_chat(**kw)
  │       └─ 同上逻辑，sleep(4s)
  │
  └─ 3. 最终兜底：再调一次 _safe_chat（不 sleep）
```
### 12.4 瞬态错误 vs 非瞬态错误
```python
_TRANSIENT_ERROR_MARKERS = (
    "429", "rate limit",           # 限流
    "500", "502", "503", "504",    # 服务端错误
    "overloaded", "timeout",       # 过载/超时
    "connection", "server error",  # 连接/服务错误
    "temporarily unavailable",     # 暂时不可用
)
```
只有包含这些关键词的错误才会重试。其他错误（如参数错误、认证失败）直接返回，不浪费时间重试。
### 12.5 图片降级（精妙设计）
```python
_IMAGE_UNSUPPORTED_MARKERS = (
    "image_url is only supported",
    "does not support image",
    "images are not supported",
    ...
)
if self._is_image_unsupported_error(response.content):
    stripped = self._strip_image_content(messages)
    if stripped is not None:
        return await self._safe_chat(**{**kw, "messages": stripped})
```
当模型不支持图片输入时（比如 GLM-5-Turbo），自动把消息中的 `image_url` 替换为 `"[image omitted]"` 文本占位符，然后重试。这样用户发图片时不会直接报错，而是得到一个"模型不支持图片"的文本回复。
### 12.6 `_safe_chat()` — 异常兜底
```python
async def _safe_chat(self, **kwargs):
    try:
        return await self.chat(**kwargs)
    except asyncio.CancelledError:
        raise  # /stop 取消，不吞
    except Exception as exc:
        return LLMResponse(content=f"Error calling LLM: {exc}", finish_reason="error")
```
所有异常（网络超时、JSON 解析错误等）都被转成 `finish_reason="error"` 的 LLMResponse。这样 `chat_with_retry` 可以统一通过 `finish_reason` 判断是否需要重试，不需要 try-catch。
### 12.7 `_sanitize_empty_content()` — 空内容清理
```python
# MCP 工具可能返回空字符串，大多数 provider 会报 400 错误
if isinstance(content, str) and not content:
    # assistant + tool_calls → content 设为 None（合法）
    # 其他 → content 设为 "(empty)"（避免 400）
```
### 12.8 `GenerationSettings` — 全局默认参数
```python
@dataclass(frozen=True)
class GenerationSettings:
    temperature: float = 0.7
    max_tokens: int = 4096
    reasoning_effort: str | None = None
```
`frozen=True` 表示不可变。存储在 provider 实例上，所有调用共享默认值。`chat_with_retry` 用 `_SENTINEL` 哨兵值区分"用户没传"和"用户传了 None"。

## 13. CronService (`cron/service.py`)
### 13.1 三种调度模式
```python
@dataclass
class CronSchedule:
    kind: str           # "at" | "every" | "cron"
    at_ms: int | None   # 一次性：指定时间戳（毫秒）
    every_ms: int | None # 间隔：每隔 N 毫秒
    expr: str | None     # cron 表达式：如 "0 9 * * 1-5"
    tz: str | None       # 时区：如 "Asia/Shanghai"
```

| 模式      | 示例                    | 用途    |
| ------- | --------------------- | ----- |
| `at`    | 2026-03-31 15:00 执行一次 | 一次性提醒 |
| `every` | 每隔 30 分钟              | 周期性检查 |
| `cron`  | 工作日早上 9 点             | 定时提醒  |
### 13.2 `_compute_next_run()` — 计算下次执行时间
```python
def _compute_next_run(schedule, now_ms):
    if schedule.kind == "at":
        return schedule.at_ms if schedule.at_ms > now_ms else None  # 过期返回 None
    if schedule.kind == "every":
        return now_ms + schedule.every_ms  # 从现在起算间隔
    if schedule.kind == "cron":
        cron = croniter(schedule.expr, base_dt)  # 用 croniter 库解析
        next_dt = cron.get_next(datetime)
        return int(next_dt.timestamp() * 1000)
```
### 13.3 定时器机制（`_arm_timer`）
CronService 不用轮询，而是用**精确睡眠**：
```python
def _arm_timer(self):
    next_wake = self._get_next_wake_ms()  # 最近的下次执行时间
    delay_s = (next_wake - now_ms) / 1000  # 计算需要睡多久
    async def tick():
        await asyncio.sleep(delay_s)  # 精确睡到下次执行时间
        if self._running:
            await self._on_timer()    # 执行到期的任务
    self._timer_task = asyncio.create_task(tick())
```
每次执行完任务后，重新计算下次执行时间，再 `_arm_timer()`。这样 CPU 开销极低——大部分时间都在 sleep。
### 13.4 `_on_timer()` — 到期执行
```python
async def _on_timer(self):
    self._load_store()  # 重新加载（支持外部修改 jobs.json）
    now = _now_ms()
    due_jobs = [j for j in jobs if j.enabled and now >= j.next_run_at_ms]
    for job in due_jobs:
        await self._execute_job(job)
    self._save_store()
    self._arm_timer()  # 重新设置定时器
```
### 13.5 `_execute_job()` — 执行单个任务
```python
async def _execute_job(self, job):
    try:
        response = await self.on_job(job)  # 回调：走 agent 循环
        job.state.last_status = "ok"
    except Exception as e:
        job.state.last_status = "error"
        job.state.last_error = str(e)
    # 一次性任务：执行后禁用或删除
    if job.schedule.kind == "at":
        if job.delete_after_run:
            self._store.jobs = [j for j in jobs if j.id != job.id]
        else:
            job.enabled = False
    else:
        # 周期任务：计算下次执行时间
        job.state.next_run_at_ms = _compute_next_run(job.schedule, _now_ms())
```
### 13.6 磁盘持久化 + 热重载
```python
def _load_store(self):
    # 检查文件修改时间
    if self._store and mtime != self._last_mtime:
        logger.info("Cron: jobs.json modified externally, reloading")
        self._store = None  # 强制重新加载
```
每次 `_on_timer()` 都会调用 `_load_store()`，如果 `jobs.json` 被外部修改（比如 agent 通过 CronTool 添加了新任务），会自动重新加载。
### 13.7 `on_job` 回调
CronService 本身不执行 agent 逻辑，而是通过回调委托：
```python
# 在 gateway 启动时注册
cron_service = CronService(store_path, on_job=handle_cron_job)
async def handle_cron_job(job):
    # 构造 InboundMessage，注入 bus
    msg = InboundMessage(
        channel=job.payload.channel or "cli",
        chat_id=job.payload.to or "direct",
        content=job.payload.message,
    )
    await bus.publish_inbound(msg)
```
这样定时任务和用户消息走同一条处理路径。

## 14. HeartbeatService (`heartbeat/service.py`)
### 14.1 与 Cron 的区别
| | Cron | Heartbeat |
|---|---|---|
| 触发方式 | 精确时间点 | 固定间隔 |
| 目的 | 提醒用户 | agent 自检 |
| 任务来源 | jobs.json | HEARTBEAT.md |
| 执行者 | 走 agent 循环 | 走 agent 循环 |
| 通知用户 | 总是通知 | 经过 Evaluator 判断 |
### 14.2 两阶段设计
```
HeartbeatService._tick()
  │
  ├─ Phase 1: _decide()
  │   读取 HEARTBEAT.md → 调 LLM（带 heartbeat 虚拟工具）
  │   → LLM 返回 action="skip" 或 "run"
  │
  ├─ 如果 skip → 结束，不打扰
  │
  └─ 如果 run → Phase 2: on_execute()
      走 agent 循环执行任务 → 得到 response
      → evaluate_response() 判断是否通知用户
      → 如果需要 → on_notify() 发送通知
```
### 14.3 Phase 1: `_decide()` — LLM 工具调用判断
```python
_HEARTBEAT_TOOL = [{
    "type": "function",
    "function": {
        "name": "heartbeat",
        "parameters": {
            "properties": {
                "action": {"enum": ["skip", "run"]},
                "tasks": {"description": "Natural-language summary of active tasks"},
            },
            "required": ["action"],
        },
    },
}]
```
**为什么用工具调用而不是自由文本？**
旧版设计让 LLM 直接输出文本，然后解析特定 token（如 `HEARTBEAT_OK`）来判断。这不可靠——LLM 可能输出格式混乱的文本。用工具调用，返回值是结构化的 JSON，解析可靠。
LLM 看到的 prompt：
```
You are a heartbeat agent. Call the heartbeat tool to report your decision.
Current Time: 2026-03-31 14:05 (Tuesday) (CST)
Review the following HEARTBEAT.md and decide whether there are active tasks.
# Heartbeat Tasks
## Active Tasks
<!-- (HEARTBEAT.md 的内容) -->
```
LLM 调用 `heartbeat(action="skip")` 或 `heartbeat(action="run", tasks="...")`。
### 14.4 Phase 2: 执行 + 通知门控
```python
if self.on_execute:
    response = await self.on_execute(tasks)  # 走 agent 循环
    if response:
        should_notify = await evaluate_response(response, tasks, ...)
        if should_notify and self.on_notify:
            await self.on_notify(response)
```
`on_execute` 回调走完整的 agent 循环（调 LLM + 执行工具），返回 agent 的文本响应。然后 `evaluate_response` 判断这个响应是否值得通知用户。

## 15. Evaluator (`utils/evaluator.py`)
### 15.1 设计动机
后台任务（Heartbeat、Cron）执行后，结果可能是：
- "一切正常，没有需要处理的事情" → **不需要通知**
- "发现了一个错误，需要你看看" → **需要通知**
如果每次都通知用户，会造成打扰。Evaluator 就是一个**通知门控**。
### 15.2 实现方式
和 Heartbeat 的 `_decide()` 一样，用 LLM 工具调用做判断：
```python
_EVALUATE_TOOL = [{
    "type": "function",
    "function": {
        "name": "evaluate_notification",
        "parameters": {
            "properties": {
                "should_notify": {"type": "boolean"},
                "reason": {"type": "string"},
            },
            "required": ["should_notify"],
        },
    },
}]
```
LLM 看到的 prompt：
```
You are a notification gate for a background agent.
Notify when the response contains actionable information, errors,
completed deliverables, or anything the user explicitly asked to
be reminded about.
Suppress when the response is a routine status check with nothing
new, a confirmation that everything is normal, or essentially empty.
## Original task
{task_context}
## Agent response
{response}
```
LLM 调用 `evaluate_notification(should_notify=True/False, reason="...")`。
### 15.3 安全兜底
```python
async def evaluate_response(...) -> bool:
    try:
        ...
        if not llm_response.has_tool_calls:
            return True  # LLM 没调工具 → 默认通知（不丢消息）
        return bool(should_notify)
    except Exception:
        return True  # 异常 → 默认通知（宁可多通知，不能漏通知）
```
**核心原则：宁可多通知，不能漏通知**。任何失败都默认 `True`（通知）。
### 15.4 轻量级调用
```python
llm_response = await provider.chat_with_retry(
    ...,
    max_tokens=256,      # 极短的输出
    temperature=0.0,     # 确定性输出
)
```
只需要一个 boolean 判断，不需要创造性输出，所以用极短的 max_tokens 和零温度。

## 五个模块的协作关系
```
用户消息
  │
  ▼
AgentLoop._process_message()
  │
  ├── spawn() → SubagentManager
  │   │  独立 Task，并行运行
  │   │  完成后 → bus.publish_inbound(system 消息)
  │   │  → 主 agent 总结 → bus.publish_outbound()
  │   │
  ├── provider.chat_with_retry() → LLMProvider
  │   │  重试瞬态错误
  │   │  图片不支持 → 自动降级
  │   │  异常 → finish_reason="error"
  │   │
  └── memory_consolidator → MemoryStore
       └── provider.chat_with_retry()（用 save_memory 工具）
CronService（独立协程）
  │  _arm_timer() → sleep → _on_timer()
  │  → on_job() → bus.publish_inbound()
  │  → AgentLoop 处理 → Evaluator 判断是否通知
HeartbeatService（独立协程）
  │  sleep(interval) → _tick()
  │  → _decide()（LLM 工具调用）
  │  → on_execute() → AgentLoop 处理
  │  → Evaluator 判断是否通知
  │  → on_notify() → bus.publish_outbound()
```
