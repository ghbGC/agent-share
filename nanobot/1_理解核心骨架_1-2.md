> **2. 事件类型** (`bus/events.py` — InboundMessage / OutboundMessage 数据结构
## 1. MessageBus (`bus/queue.py`)
### 本质
MessageBus 就是**两个 asyncio.Queue**，总共 44 行：
```python
class MessageBus:
    def __init__(self):
        self.inbound: asyncio.Queue[InboundMessage]   # 用户 → agent
        self.outbound: asyncio.Queue[OutboundMessage]  # agent → 用户
```
### 为什么需要它？
没有 MessageBus 的话，AgentLoop 要直接调用 `feishu_channel.send()`，Channel 要直接调用 `agent_loop.process()`，两者紧耦合。有了 MessageBus：
```
Channel ──publish_inbound──→ [inbound queue] ──consume_inbound──→ AgentLoop
AgentLoop ──publish_outbound─→ [outbound queue] ──consume_outbound─→ Channel
```
两边互不知道对方的存在，加新 Channel 不用改 AgentLoop 代码。
### 四个方法
```python
publish_inbound(msg)   # Channel 调用：把用户消息放入 inbound 队列
consume_inbound()       # AgentLoop 调用：从 inbound 队列取消息（队空时阻塞等待）
publish_outbound(msg)  # AgentLoop 调用：把响应放入 outbound 队列
consume_outbound()     # ChannelManager 调用：从 outbound 队列取响应
```
`asyncio.Queue` 的 `get()` 在队列为空时会挂起当前协程（不占 CPU），有消息放入时自动唤醒。这就是异步编程中"生产者-消费者"模式的标准实现。
### 谁在消费 outbound？
取决于运行模式：
- **Gateway 模式**：`ChannelManager._dispatch_outbound()` 协程不断从 outbound 取消息，根据 `msg.channel` 路由到对应 Channel 的 `send()` 方法
- **CLI 模式**：自己写一个 `_consume_outbound()` 协程，直接打印到终端

## 2. 事件类型 (`bus/events.py`)
### InboundMessage — 入站消息（用户 → agent）
```python
@dataclass
class InboundMessage:
    channel: str              # 来源通道："feishu", "telegram", "cli", "system"
    sender_id: str           # 发送者 ID
    chat_id: str             # 会话 ID
    content: str             # 消息文本
    timestamp: datetime       # 时间戳（自动生成）
    media: list[str]         # 附件路径列表（图片、音频等）
    metadata: dict           # 通道特有的元数据（如飞书的 message_id）
    session_key_override: str | None  # 可选的会话 key 覆盖
    @property
    def session_key(self) -> str:
        return self.session_key_override or f"{self.channel}:{self.chat_id}"
```
**`session_key` 是关键**——它决定了用哪个 Session 存储对话历史。默认格式 `"channel:chat_id"`，比如：
- `"feishu:ou_xxx"`
- `"telegram:xxx"`
- `"cli:direct"`
`session_key_override` 用于线程级会话（比如飞书群的每个帖子用不同 session）。
**`channel="system"` 是特殊值**——子 agent 完成任务后，通过 `channel="system"` 注入消息，`chat_id` 编码了原始路由信息（如 `"feishu:ou_xxx"`），主 agent 解析后恢复原始通道。
### OutboundMessage — 出站消息（agent → 用户）
```python
@dataclass
class OutboundMessage:
    channel: str              # 目标通道
    chat_id: str             # 目标会话 ID
    content: str             # 消息文本
    reply_to: str | None     # 回复的消息 ID（用于引用回复）
    media: list[str]         # 附件路径列表
    metadata: dict           # 元数据（如 `_progress=True` 表示进度消息）
```
**`metadata` 的用途**：
- `_progress: True` — 标记为进度消息（飞书以卡片形式展示）
- `_tool_hint: True` — 标记为工具调用提示（如 `read_file("path")`）
- `_progress: False` + `_tool_hint: False` — 普通消息
- `message_id` — 原始消息 ID（用于回复引用）
### 两个数据结构的关系
```
用户发 "你好"
  │
  ▼
InboundMessage(
    channel="feishu",
    sender_id="ou_xxx",
    chat_id="ou_xxx",
    content="你好",
    media=[],
    metadata={"message_id": "om_xxx"},
)
  │
  ▼  (经过 AgentLoop 处理)
  │
OutboundMessage(
    channel="feishu",
    chat_id="ou_xxx",
    content="你好！有什么可以帮你的？",
    media=[],
    metadata={},
)
```
InboundMessage 携带**输入信息**（谁发的、从哪发的、附件），OutboundMessage 携带**输出信息**（发给谁、发什么、附件）。两者通过 MessageBus 的两个队列连接。
