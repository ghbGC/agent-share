## 23. 配置加载 (`config/loader.py` + `config/schema.py` + `config/paths.py`)
### 23.1 整体架构
配置系统分三层：
```
config.json（磁盘文件）
    │
    ▼
loader.py（加载 + 迁移）
    │
    ▼
schema.py（Pydantic 模型定义 + 验证）
    │
    ▼
paths.py（路径派生）
```
### 23.2 `loader.py` — 加载与迁移
**配置路径解析**：
```python
_current_config_path: Path | None = None  # 全局变量，支持多实例
def get_config_path():
    if _current_config_path:
        return _current_config_path
    return Path.home() / ".nanobot" / "config.json"  # 默认路径
```
`set_config_path()` 允许在启动时通过 `--config` 参数指定配置文件路径。路径存储在全局变量中，后续所有路径派生都基于它。
**加载流程**：
```python
def load_config(config_path=None):
    path = config_path or get_config_path()
    if path.exists():
        data = json.load(f)
        data = _migrate_config(data)       # 版本迁移
        return Config.model_validate(data)  # Pydantic 验证
    return Config()  # 文件不存在 → 返回默认配置
```
**保存流程**：
```python
def save_config(config, config_path=None):
    data = config.model_dump(by_alias=True)  # camelCase 输出
    json.dump(data, f, indent=2, ensure_ascii=False)
```
`by_alias=True` 确保输出用 camelCase（如 `maxTokens`），而不是 Python 的 snake_case（如 `max_tokens`）。
**配置迁移**：
```python
def _migrate_config(data):
    # 旧版: tools.exec.restrictToWorkspace
    # 新版: tools.restrictToWorkspace
    if "restrictToWorkspace" in exec_cfg and "restrictToWorkspace" not in tools:
        tools["restrictToWorkspace"] = exec_cfg.pop("restrictToWorkspace")
```
简单的字段搬家，保证旧配置文件能被新版兼容读取。
### 23.3 `schema.py` — Pydantic 模型（核心）
**Base 类 — camelCase 兼容**：
```python
class Base(BaseModel):
    model_config = ConfigDict(alias_generator=to_camel, populate_by_name=True)
```
`alias_generator=to_camel` 让 Pydantic 同时接受 `maxTokens`（camelCase）和 `max_tokens`（snake_case）两种写法。`populate_by_name=True` 允许用字段名填充。
**Config 根结构**：
```python
class Config(BaseSettings):
    agents: AgentsConfig       # agent 配置（模型、workspace 等）
    channels: ChannelsConfig   # 通道配置（飞书、Telegram 等）
    providers: ProvidersConfig # LLM 提供者配置
    gateway: GatewayConfig     # 网关配置（端口、心跳等）
    tools: ToolsConfig        # 工具配置（exec、web、mcp 等）
```
**嵌套结构**：
```
Config
├── agents
│   └── defaults
│       ├── workspace: "~/.nanobot/workspace"
│       ├── model: "anthropic/claude-opus-4-5"
│       ├── provider: "auto"
│       ├── max_tokens: 8192
│       ├── context_window_tokens: 65536
│       ├── temperature: 0.1
│       ├── max_tool_iterations: 40
│       └── reasoning_effort: None
├── channels
│   ├── send_progress: True
│   ├── send_tool_hints: False
│   ├── feishu: { enabled, app_id, app_secret, ... }  # extra="allow"
│   ├── telegram: { enabled, bot_token, ... }
│   └── ...
├── providers
│   ├── custom: { api_key, api_base, extra_headers }
│   ├── anthropic: { api_key, api_base }
│   ├── openai: { api_key, api_base }
│   ├── zhipu: { api_key, api_base, extra_headers }
│   ├── deepseek: { api_key, api_base }
│   └── ...（20+ 提供者）
├── gateway
│   ├── host: "0.0.0.0"
│   ├── port: 18790
│   └── heartbeat: { enabled, interval_s: 1800 }
└── tools
    ├── restrict_to_workspace: False
    ├── web
    │   ├── proxy: None
    │   └── search: { provider, api_key, max_results }
    ├── exec: { timeout: 60, path_append: "" }
    └── mcp_servers: { "server_name": { type, command, url, ... } }
```
**ChannelsConfig 的 `extra="allow"`**：
```python
class ChannelsConfig(Base):
    model_config = ConfigDict(extra="allow")
```
Pydantic 默认不允许未定义的字段。`extra="allow"` 让它可以接受任意额外的字段（如 `feishu`、`telegram`），每个通道在 `__init__` 中自己解析。
**环境变量覆盖**：
```python
class Config(BaseSettings):
    model_config = ConfigDict(env_prefix="NANOBOT_", env_nested_delimiter="__")
```
`BaseSettings` 是 pydantic-settings 的类，支持从环境变量读取配置。`env_prefix="NANOBOT_"` + `env_nested_delimiter="__"` 意味着：
```bash
# 等价于 config.json 中的 agents.defaults.model
export NANOBOT_AGENTS__DEFAULTS__MODEL="deepseek/deepseek-chat"
```
优先级：环境变量 > config.json > 默认值。
### 23.4 `_match_provider()` — 模型→提供者匹配（重点）
这是配置系统中最复杂的逻辑。给定一个模型名（如 `"deepseek/deepseek-chat"`），找到对应的 provider 配置：
```python
def _match_provider(self, model=None):
    # 1. 如果 provider 被强制指定（非 "auto"），直接用
    forced = self.agents.defaults.provider
    if forced != "auto":
        p = getattr(self.providers, forced, None)
        return (p, forced) if p else (None, None)
    # 2. 从模型名提取前缀，精确匹配
    model_lower = "deepseek/deepseek-chat"
    model_prefix = "deepseek"
    # → 匹配 providers.deepseek
    # 3. 关键词匹配（按 PROVIDERS 注册顺序）
    for spec in PROVIDERS:
        if any(kw in model_lower for kw in spec.keywords):
            if spec.is_oauth or spec.is_local or p.api_key:
                return p, spec.name
    # 4. 本地 provider 兜底（Ollama 等）
    # 5. 通用兜底（第一个有 api_key 的 provider）
```
**匹配优先级**：
1. 强制指定的 provider（`provider: "deepseek"`）
2. 模型名前缀精确匹配（`deepseek/xxx` → `providers.deepseek`）
3. 关键词匹配（按注册顺序）
4. 本地 provider（Ollama 等，按 api_base 关键词匹配）
5. 通用兜底（第一个有 api_key 的非 OAuth provider）
### 23.5 `paths.py` — 路径派生
所有路径都基于 `config.json` 的位置派生：
```python
def get_data_dir():
    return ensure_dir(get_config_path().parent)
    # config.json 在 ~/.nanobot/config.json → data_dir = ~/.nanobot/
def get_runtime_subdir(name):
    return ensure_dir(get_data_dir() / name)
    # get_runtime_subdir("media") → ~/.nanobot/media/
def get_cron_dir():
    return get_runtime_subdir("cron")  # ~/.nanobot/cron/
def get_workspace_path(workspace=None):
    return ensure_dir(Path(workspace or "~/.nanobot/workspace"))
```
**设计意图**：支持多实例。如果 `--config /path/to/config.json`，所有数据目录都在 `/path/to/` 下，不会和默认实例冲突。

## 24. 辅助函数 (`utils/helpers.py`)
### 24.1 图片 MIME 检测
```python
def detect_image_mime(data: bytes) -> str | None:
    if data[:8] == b"\x89PNG\r\n\x1a\n":   return "image/png"
    if data[:3] == b"\xff\xd8\xff":          return "image/jpeg"
    if data[:6] in (b"GIF87a", b"GIF89a"): return "image/gif"
    if data[:4] == b"RIFF" and data[8:12] == b"WEBP": return "image/webp"
    return None
```
通过文件头的 magic bytes（魔数）检测图片类型，不依赖文件扩展名。这比 `mimetypes.guess_type()` 更可靠——用户可能把 `.jpg` 文件重命名为 `.txt`。
### 24.2 Token 估算（三级降级）
**第一级：`estimate_prompt_tokens_chain()`** — provider 自带计数器
```python
def estimate_prompt_tokens_chain(provider, model, messages, tools):
    provider_counter = getattr(provider, "estimate_prompt_tokens", None)
    if callable(provider_counter):
        tokens, source = provider_counter(messages, tools, model)
        if tokens > 0:
            return tokens, str(source)
```
如果 provider（如 litellm）自带 token 计数方法，优先使用，最准确。
**第二级：`estimate_prompt_tokens()`** — tiktoken 估算
```python
def estimate_prompt_tokens(messages, tools=None):
    enc = tiktoken.get_encoding("cl100k_base")
    parts = [msg["content"] for msg in messages if isinstance(msg["content"], str)]
    if tools:
        parts.append(json.dumps(tools))
    return len(enc.encode("\n".join(parts)))
```
用 tiktoken（OpenAI 的 tokenizer）估算。`cl100k_base` 是 GPT-4 用的编码器，对其他模型的估算会有偏差，但作为近似值够用。
**第三级：`estimate_message_tokens()`** — 粗略估算
```python
def estimate_message_tokens(message):
    # 提取所有文本内容
    # 如果 tiktoken 失败，用 len(payload) // 4 作为粗略估算
    return max(1, len(payload) // 4)
```
英文平均 4 个字符 ≈ 1 个 token，中文约 1-2 个字符 ≈ 1 个 token，所以 `// 4` 是保守估算。
### 24.3 `build_assistant_message()` — 构造 assistant 消息
```python
def build_assistant_message(content, tool_calls, reasoning_content, thinking_blocks):
    msg = {"role": "assistant", "content": content}
    if tool_calls:
        msg["tool_calls"] = tool_calls
    if reasoning_content is not None:
        msg["reasoning_content"] = reasoning_content  # DeepSeek-R1 的思考链
    if thinking_blocks:
        msg["thinking_blocks"] = thinking_blocks      # Anthropic 的扩展思考
    return msg
```
统一构造 assistant 消息，兼容不同 provider 的扩展字段。
### 24.4 `split_message()` — 消息分片
```python
def split_message(content, max_len=2000):
    # 优先在换行处切分
    pos = cut.rfind('\n')
    # 其次在空格处切分
    if pos <= 0:
        pos = cut.rfind(' ')
    # 最后硬切
    if pos <= 0:
        pos = max_len
```
用于 Discord 等有消息长度限制的通道。优先保持语义完整（不在单词中间切断）。
