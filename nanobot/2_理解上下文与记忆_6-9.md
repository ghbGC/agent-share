>**6. ContextBuilder** (`agent/context.py`) — system prompt 拼接逻辑，理解各部分优先级
>**7. MemoryStore** (`agent/memory.py` 上半部分) — 两层记忆读写 + `consolidate()` 用 LLM 做总结
>**8. MemoryConsolidator** (`agent/memory.py` 下半部分) — token 估算 + 边界选择 + 循环压缩策略
>**9. SkillsLoader** (`agent/skills.py`) — 技能发现（workspace > builtin）、frontmatter 解析、依赖检查
## 6. ContextBuilder (`agent/context.py`)
### 6.1 整体定位
ContextBuilder 是 nanobot 的**提示词组装工厂**。每次 LLM 调用前，它负责把所有碎片拼成一个完整的消息列表。它是 agent 的"大脑外壳"——决定了 LLM 看到的全部上下文。
### 6.2 `build_system_prompt()` — system prompt 拼接（核心）
system prompt 由 **5 个部分**按固定顺序拼接，用 `\n\n---\n\n` 分隔：
```
┌─────────────────────────────────────────┐
│ 1. Identity（身份）                      │  ← 代码硬编码，不可配置
│    "You are nanobot, a helpful AI..."   │
├─────────────────────────────────────────┤
│ 2. Bootstrap Files（引导文件）            │  ← workspace 下的 AGENTS.md 等
│    AGENTS.md + SOUL.md + USER.md + ...  │
├─────────────────────────────────────────┤
│ 3. Memory（长期记忆）                     │  ← MEMORY.md 内容
│    ## Long-term Memory                  │
│    ...                                  │
├─────────────────────────────────────────┤
│ 4. Active Skills（自动加载的技能）         │  ← always=true 的技能全文
│    ### Skill: memory                    │
│    ...                                  │
├─────────────────────────────────────────┤
│ 5. Skills Summary（技能摘要）             │  ← 所有技能的 XML 摘要
│    <skills>                             │
│      <skill available="true">           │
│        <name>...</name>                 │
│        <description>...</description>   │
│      </skill>                           │
│    </skills>                            │
└─────────────────────────────────────────┘
```
**优先级设计**：越靠前越"基础"，越靠后越"可选"。Identity 是不可变的，Bootstrap Files 是用户配置的，Memory 是动态更新的，Skills 是按需加载的。
### 6.3 `_get_identity()` — 身份部分
硬编码的 nanobot 身份信息，包含：

| 字段 | 来源 | 说明 |
|------|------|------|
| Runtime | `platform.system()` + `platform.python_version()` | 告诉 LLM 当前运行环境 |
| Workspace | `self.workspace` 路径 | 告诉 LLM 工作目录在哪 |
| Platform Policy | 根据 OS 选择 | POSIX vs Windows 的行为差异 |
| Guidelines | 硬编码 | 5 条行为准则（先读后写、不预测结果等） |
**设计意图**：让 LLM 知道"我是谁、在哪运行、该遵守什么规则"，不需要用户每次重复说明。
### 6.4 `_load_bootstrap_files()` — 引导文件
```python
BOOTSTRAP_FILES = ["AGENTS.md", "SOUL.md", "USER.md", "TOOLS.md"]
```
按顺序从 workspace 根目录读取，存在就加载，不存在就跳过。这就是你平时编辑的那几个文件——它们直接决定了 agent 的性格、用户偏好、工具说明。
### 6.5 `build_messages()` — 构建完整消息列表
```python
def build_messages(self, history, current_message, ...):
    return [
        {"role": "system", "content": self.build_system_prompt(...)},  # 系统提示
        *history,                                                       # 历史对话
        {"role": "user", "content": merged},                            # 当前用户消息
    ]
```
**关键细节：runtime context 注入**
```python
runtime_ctx = self._build_runtime_context(channel, chat_id)
# → "[Runtime Context — metadata only, not instructions]\nCurrent Time: ...\nChannel: ..."
```
这个 runtime context 被合并到**用户消息**里（不是 system prompt），原因在注释里写了：
> Merge runtime context and user content into a single user message to avoid consecutive same-role messages that some providers reject.
有些 LLM provider 不允许连续两条相同 role 的消息。如果 runtime context 单独作为一条 user 消息，而紧接着又是真正的 user 消息，就会触发这个限制。
**另一个细节：`_RUNTIME_CONTEXT_TAG`**
```
[Runtime Context — metadata only, not instructions]
```
这个标签告诉 LLM："这只是元数据，不是给你的指令"。防止 LLM 把 "Channel: feishu" 当成某种指令去执行。
### 6.6 `_build_user_content()` — 图片处理
```python
def _build_user_content(self, text, media):
    # 如果有图片 → 转成 base64 的 image_url 格式
    # 如果没有图片 → 直接返回文本
```
图片以 `data:image/png;base64,...` 格式内联到消息中，这是 OpenAI 多模态消息的标准格式。
### 6.7 `add_tool_result()` / `add_assistant_message()` — 循环中追加消息
这两个方法在 agent 循环中使用，每次工具调用后追加 tool result，每次 LLM 响应后追加 assistant message。它们直接修改传入的 `messages` 列表（mutate in place）并返回。

## 7. MemoryStore (`agent/memory.py` 上半部分)
### 7.1 两层记忆架构
```
MEMORY.md（长期记忆）          HISTORY.md（历史日志）
├── 用户偏好                   ├── [2026-03-21 16:00] USER: ...
├── 项目上下文                 ├── [2026-03-21 16:05] ASSISTANT [tools: read_file]: ...
├── 重要事实                   ├── [2026-03-22 10:00] USER: ...
└── 每次 consolidate 更新      └── 只追加，不修改
```
- **MEMORY.md**：每次 consolidate 时**全量覆盖**（LLM 返回完整的新版本）
- **HISTORY.md**：每次 consolidate 时**追加**一条摘要
### 7.2 `consolidate()` — 用 LLM 做总结（核心方法）
这是整个记忆系统最精巧的部分。流程如下：
```
consolidate(messages, provider, model)
  │
  ├─ 1. 读取当前 MEMORY.md
  │
  ├─ 2. 构造 prompt：
  │     "你是一个记忆整理 agent，请处理以下对话，调用 save_memory 工具"
  │     + 当前 MEMORY.md 内容
  │     + 待处理的对话消息
  │
  ├─ 3. 调 LLM，强制 tool_choice=save_memory
  │
  ├─ 4. 处理 LLM 返回的 tool call：
  │     ├─ history_entry → 追加到 HISTORY.md
  │     └─ memory_update → 如果有变化，覆盖 MEMORY.md
  │
  └─ 5. 失败处理 → _fail_or_raw_archive()
```
**为什么用 tool call 而不是直接让 LLM 输出文本？**
因为 tool call 有结构化的参数（`history_entry` 和 `memory_update`），比自由文本输出更可靠。LLM 可能输出格式混乱的文本，但 tool call 的参数是 JSON，容易解析。
**`_SAVE_MEMORY_TOOL` 的设计**：
```python
_SAVE_MEMORY_TOOL = [{
    "type": "function",
    "function": {
        "name": "save_memory",
        "parameters": {
            "properties": {
                "history_entry": "一段摘要，以 [YYYY-MM-DD HH:MM] 开头",
                "memory_update": "完整更新后的长期记忆 markdown",
            },
            "required": ["history_entry", "memory_update"],
        },
    },
}]
```
注意 `memory_update` 要求返回**完整的新版本**（不是增量），这样即使 LLM "忘记"了旧内容，也不会丢失信息。
### 7.3 `tool_choice` 兼容性处理
```python
forced = {"type": "function", "function": {"name": "save_memory"}}
response = await provider.chat_with_retry(..., tool_choice=forced)
# 如果 provider 不支持强制 tool_choice（比如某些模型）
if response.finish_reason == "error" and _is_tool_choice_unsupported(response.content):
    response = await provider.chat_with_retry(..., tool_choice="auto")  # 降级为 auto
```
这是一个兼容性设计：先用强制模式确保 LLM 一定调用 `save_memory`，如果 provider 报错不支持，就降级为 `"auto"` 模式。
### 7.4 `_fail_or_raw_archive()` — 降级策略
```python
_MAX_FAILURES_BEFORE_RAW_ARCHIVE = 3
def _fail_or_raw_archive(self, messages):
    self._consecutive_failures += 1
    if self._consecutive_failures < 3:
        return False  # 告诉调用方"归档失败"
    self._raw_archive(messages)  # 连续失败 3 次，直接 dump 原始消息
    self._consecutive_failures = 0
    return True
```
**三级降级**：
1. 正常：LLM 总结 → 结构化写入
2. LLM 没调用 tool / 参数错误 → 返回 False（让调用方重试）
3. 连续失败 3 次 → `_raw_archive()` 直接把原始消息格式化后追加到 HISTORY.md
这保证了**即使 LLM 完全不配合，对话记录也不会丢失**。
### 7.5 `_format_messages()` — 消息格式化
```python
@staticmethod
def _format_messages(messages):
    for message in messages:
        tools = f" [tools: {', '.join(message['tools_used'])}]" if message.get("tools_used") else ""
        lines.append(f"[{timestamp}] {ROLE}{tools}: {content}")
```
把结构化的消息列表转成人类可读的文本格式，供 LLM 阅读和总结。注意 `tools_used` 字段——这是 Session.add_message 时通过 `**kwargs` 传入的，记录了这轮对话用了哪些工具。

## 8. MemoryConsolidator (`agent/memory.py` 下半部分)
### 8.1 与 MemoryStore 的分工
| | MemoryStore | MemoryConsolidator |
|---|---|---|
| 职责 | 读写文件 + 调 LLM 做总结 | 决定**何时**总结 + **总结多少** |
| 关注点 | 单次归档的质量 | 整体 token 预算管理 |
| 类比 | "怎么写总结" | "什么时候写总结" |
### 8.2 `maybe_consolidate_by_tokens()` — 核心调度方法
这是在每次消息处理完后被调用的，流程如下：
```
maybe_consolidate_by_tokens(session)
  │
  ├─ 1. 获取当前 session 的锁（防止并发归档）
  │
  ├─ 2. 估算当前 prompt 的 token 数
  │     estimated = estimate_session_prompt_tokens(session)
  │
  ├─ 3. 如果 estimated < context_window_tokens → 不需要压缩，返回
  │
  ├─ 4. 进入循环（最多 5 轮）：
  │     │
  │     ├─ 目标：把 token 降到 context_window_tokens / 2
  │     │
  │     ├─ pick_consolidation_boundary() → 找到要归档的消息边界
  │     │
  │     ├─ 取出消息块 → consolidate_messages() → LLM 总结
  │     │
  │     ├─ 更新 session.last_consolidated → 保存 session
  │     │
  │     └─ 重新估算 token 数 → 如果达标就退出循环
  │
  └─ 5. 释放锁
```
**为什么目标是 `context_window_tokens / 2`？**
因为 system prompt + 工具定义本身就占了不少 token。如果只压缩到接近上限，下一轮对话加上新消息就可能超限。留一半余量更安全。
### 8.3 `estimate_session_prompt_tokens()` — token 估算
```python
def estimate_session_prompt_tokens(self, session):
    history = session.get_history(max_messages=0)  # 取全部未归档消息
    probe_messages = self._build_messages(
        history=history,
        current_message="[token-probe]",  # 假消息，用于估算
        channel=channel,
        chat_id=chat_id,
    )
    return estimate_prompt_tokens_chain(
        self.provider, self.model, probe_messages, self._get_tool_definitions(),
    )
```
构造一个"假"的完整消息列表（包含 system prompt + history + 假用户消息 + 工具定义），然后调用 provider 的 token 估算方法。`max_messages=0` 在 `get_history` 里实际上会返回所有消息（因为 `-0` 在 Python 里等于取全部）。
### 8.4 `pick_consolidation_boundary()` — 选择归档边界（重点）
```python
def pick_consolidation_boundary(self, session, tokens_to_remove):
    start = session.last_consolidated
    removed_tokens = 0
    last_boundary = None
    for idx in range(start, len(session.messages)):
        message = session.messages[idx]
        if idx > start and message["role"] == "user":
            last_boundary = (idx, removed_tokens)
            if removed_tokens >= tokens_to_remove:
                return last_boundary  # 找到足够大的边界
        removed_tokens += estimate_message_tokens(message)
    return last_boundary  # 没达到目标，返回最后一个边界
```
**核心思想：在 user 消息处切割**。
为什么？因为一轮对话是 "user → assistant → tool → tool → assistant → user → ..."，在 user 消息处切割可以保证不把一轮对话从中间截断。
```
messages[last_consolidated:] = 
  [user, assistant, tool, assistant, user, assistant, tool, user, assistant]
                                         ↑
                                    在这里切割（user 消息处）
                                    归档 [user, assistant, tool, assistant]
```
返回值是 `(end_idx, removed_tokens)`，`end_idx` 是切割位置的索引。
### 8.5 锁机制
```python
self._locks: weakref.WeakValueDictionary[str, asyncio.Lock] = weakref.WeakValueDictionary()
def get_lock(self, session_key):
    return self._locks.setdefault(session_key, asyncio.Lock())
```
用 `WeakValueDictionary` 存储锁——当没有人在用某个 session 的锁时，它会被自动垃圾回收，不会内存泄漏。每个 session 一个锁，防止同一个 session 的多次归档操作并发执行。

## 9. SkillsLoader (`agent/skills.py`)
### 9.1 技能的目录结构
```
workspace/skills/              ← 用户安装的技能（高优先级）
  kimi-websearch/SKILL.md
  image_generate/SKILL.md
  camera/SKILL.md
  tts/SKILL.md
nanobot/skills/                ← 内置技能（低优先级）
  memory/SKILL.md
  cron/SKILL.md
  weather/SKILL.md
  ...
```
每个技能就是一个目录，里面必须有一个 `SKILL.md` 文件。
### 9.2 `list_skills()` — 技能发现
```python
def list_skills(self, filter_unavailable=True):
    skills = []
    # 1. 先扫 workspace/skills/
    # 2. 再扫 builtin/skills/（同名跳过，workspace 优先）
    # 3. 如果 filter_unavailable=True，过滤掉依赖不满足的
    return skills
```
**workspace 优先**：如果用户安装了一个叫 `memory` 的技能，它会覆盖内置的 `memory` 技能。
### 9.3 Frontmatter 解析
SKILL.md 文件格式：
```markdown
---
name: tts
description: "文本转语音工具..."
metadata:
  openclaw:
    emoji: 🔊
    requires:
      env:
        - API_KEY
---
# TTS / 文本转语音
...（正文内容）
```
`get_skill_metadata()` 用简单的正则解析 YAML frontmatter：
```python
if content.startswith("---"):
    match = re.match(r"^---\n(.*?)\n---", content, re.DOTALL)
    for line in match.group(1).split("\n"):
        if ":" in line:
            key, value = line.split(":", 1)
            metadata[key.strip()] = value.strip().strip('"\'')
```
注意这是**简易 YAML 解析器**，不支持嵌套结构。对于嵌套的 `metadata` 字段，它会把整个 JSON 字符串存为一个值，后续由 `_parse_nanobot_metadata()` 用 `json.loads()` 解析。
### 9.4 `_check_requirements()` — 依赖检查
```python
def _check_requirements(self, skill_meta):
    requires = skill_meta.get("requires", {})
    for b in requires.get("bins", []):      # 检查命令是否存在
        if not shutil.which(b):
            return False
    for env in requires.get("env", []):     # 检查环境变量是否设置
        if not os.environ.get(env):
            return False
    return True
```
两种依赖类型：
- `bins`：需要某个命令在 PATH 中（如 `gh`、`tmux`）
- `env`：需要某个环境变量（如 `API_KEY`）
### 9.5 `build_skills_summary()` — 技能摘要（渐进式加载）
```python
def build_skills_summary(self):
    all_skills = self.list_skills(filter_unavailable=False)  # 列出所有，包括不可用的
    # 生成 XML 格式摘要
```
输出示例：
```xml
<skills>
  <skill available="true">
    <name>tts</name>
    <description>文本转语音工具...</description>
    <location>.../skills/tts/SKILL.md</location>
  </skill>
  <skill available="false">
    <name>github</name>
    <description>Interact with GitHub...</description>
    <location>.../skills/github/SKILL.md</location>
    <requires>CLI: gh</requires>
  </skill>
</skills>
```
**渐进式加载设计**：摘要只包含名称、描述、路径、是否可用，不包含正文。LLM 看到摘要后，如果需要用某个技能，再用 `read_file` 工具读取完整的 SKILL.md。这样节省了大量 token。
### 9.6 `get_always_skills()` — 自动加载的技能
```python
def get_always_skills(self):
    for s in self.list_skills(filter_unavailable=True):
        meta = self.get_skill_metadata(s["name"]) or {}
        skill_meta = self._parse_nanobot_metadata(meta.get("metadata", ""))
        if skill_meta.get("always") or meta.get("always"):
            result.append(s["name"])
    return result
```
如果技能的 frontmatter 或 metadata JSON 中有 `always: true`，它的**完整正文**会被直接注入 system prompt（而不是只显示摘要）。目前 `memory` 技能就是 `always=true`。
### 9.7 `_strip_frontmatter()` — 去掉 YAML 头
```python
def _strip_frontmatter(self, content):
    if content.startswith("---"):
        match = re.match(r"^---\n.*?\n---\n", content, re.DOTALL)
        if match:
            return content[match.end():].strip()
    return content
```
当技能被注入 system prompt 时，YAML frontmatter 会被去掉，只保留正文。因为 frontmatter 是给 nanobot 解析用的元数据，LLM 不需要看。

## 三个模块的协作关系
```
ContextBuilder.build_system_prompt()
  │
  ├── MemoryStore.get_memory_context()     ← 读取 MEMORY.md
  │
  ├── SkillsLoader.get_always_skills()     ← 找到 always=true 的技能
  │   └── SkillsLoader.load_skills_for_context()  ← 读取完整 SKILL.md
  │
  └── SkillsLoader.build_skills_summary()  ← 生成所有技能的 XML 摘要
MemoryConsolidator.maybe_consolidate_by_tokens()
  │
  ├── MemoryStore.consolidate()            ← 调 LLM 做总结
  │   ├── 写入 HISTORY.md（追加）
  │   └── 写入 MEMORY.md（覆盖）
  │
  └── Session.last_consolidated += N       ← 更新归档偏移量
```
